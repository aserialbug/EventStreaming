# Описание форматов событий протокола Event Streaming

Все события должны соотвветствовать спецификации [CloudEvents](https://github.com/cloudevents/spec/blob/v1.0.2/cloudevents/spec.md). События рассматриваются с точки зрения источника данных - сервиса, который владеет набором данных и распространяет его обновления в виде событий.

Согласно спецификации каждое событие должно содержать следующие общие атрибуты:

|                 |                                            |              |           |                                                                                                                         |
| --------------- | ------------------------------------------ | ------------ | --------- | ----------------------------------------------------------------------------------------------------------------------- |
| Атрибут         | Тип                                        | Обязательный | Валидация | Комментарии                                                                                                             |
| id              | Строка                                     | Да           |           | Идентификатор события. Вместе с source образуют глобально-уникальный идентификатор события. Используется Guid версии 7. |
| source          | Относительный URI                          | Да           |           | Источник события. Вместе с id образуют глобально-уникальный идентификатор события. URI севриса, опубликовавшего событие |
| specversion     | Строка                                     | Да           |           | Версия спецификации. Константа, сейчас "1.0"                                                                            |
| type            | Строка                                     | Да           |           | Тип события. Имя агрегата начиная с DNS домена и имени сервиса                                                          |
| datacontenttype | Строка                                     | Нет          |           | Тип содержимого атрибута data. По сути формат полезной нагрузки события                                                 |
| dataschema      | Абсолютный URI                             | Нет          |           | Ссылка на схему данных находящихся в атрибуте data                                                                      |
| subject         | Строка                                     | Нет          |           | Тип события regular, init, control.in, control.out                                                                      |
| time            | ВремннАя метка                             | Нет          |           | Дата/время создания события. Формат?                                                                                    |
| data            | <Данные типа указанного в datacontenttype> | Нет          |           | Полезная нагрузка. В общем случае - байтовая строка                                                                     |

Пример события в виде JSON

```json
{\
"id": "D5FD311B-6FEA-430D-8B74-F33952B18205",\
"source": "https://youla.2gis.ru/integrationservice",\
"specversion": "1.0",\
"type": "ru.2gis.youla.vorwand.v1",\
"datacontenttype": "text/xml",\
"dataschema": "",\
"subject": "regular",\
"time": "2018-04-05T17:31:00Z",\
"data": "<much wow=\"xml\"/>"\
}
```

## Управляющие события

### Входящие

Событие с полем subject = [control.in](https://control.in)

Нужно предусмотреть событие отмены/остановки выгрузки данных, на случай если клиент не может их больше принять и переварить.

Если сервис-источник шардированный, то входящее сообщение должен получть каждый самостоятельный инстанс.

Если для выгрузки доступно более одной секции данных(шарда), то клиет может запросить выгрузку конкретного набора (шарда). Клиент запрашивает выгрузки секций по очереди, когда станет готов к обработке следующей секции.

### Исходящие

Событие с полем subject = control.out

Если набор данных секционированный (шардированный), то сервис должен отправить в ответ колличество секций, доступных к выгрузке. Если за каждую секцию отвечает отдельный инстанс сервиса, то каждый инстанс отвечает сообщением с идентификатором своей реплики.

## События данных

### Инициализационное

Событие с полем subject = init

### Регулярное

Событие с полем subject = regular
